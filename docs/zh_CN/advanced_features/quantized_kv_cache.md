# 量化 KV 缓存

量化 KV 缓存通过使用更低精度的数据类型（FP8 或 FP4）代替默认的 BF16 模型精度，减少键值缓存存储的内存占用。在自回归生成过程中，LLM 缓存先前计算的键值对以避免冗余计算。KV 缓存通常消耗 GPU 内存的很大一部分，尤其是对于长序列。

量化 KV 缓存是一种内存优化技术，主要通过允许缓存更多 Token 来提升吞吐量，但根据所使用的量化格式，可能会引入极小的精度下降。

```{warning}
**性能警告**：当量化 KV 缓存必须在注意力操作之前进行反量化时，如果反量化未与注意力内核融合，性能可能会极其缓慢。请始终验证您选择的注意力后端是否支持量化 KV 缓存。不支持融合的后端可能会经历显著的吞吐量下降，可能会抵消内存节省带来的收益。

**后端支持**：并非所有注意力后端都支持量化 KV 缓存。请参阅 [注意力后端](attention_backend.md) 了解哪些后端支持此功能。
```

## 支持的格式

SGLang 支持以下量化 KV 缓存格式：

### FP8 格式

[OCP（开放计算项目）](https://www.opencompute.org) 定义了两种常见的 8 位浮点格式：

- **E5M2**（5 位指数，2 位尾数）：更大的动态范围（±57344.0），精度较低
- **E4M3**（4 位指数，3 位尾数）：更高的精度，更小的动态范围（±240.0）

### FP4 格式

```{warning}
FP4 量化目前处于实验阶段。
```

[OCP（开放计算项目）](https://www.opencompute.org) 定义了 MXFP4（微缩放 FP4），一种 4 位浮点格式：

- **E2M1**（1 位符号，2 位指数，1 位尾数）：使用基于块的微缩放，其中张量被分成连续元素的块，每个块共享一个 8 位指数缩放因子。虽然 OCP 规定块大小为 32 个元素，但 SGLang 当前的 KV 缓存量化实现使用 16 个元素的块。

## 使用方法

### 启用量化 KV 缓存

要启用量化 KV 缓存，请在启动服务器时使用 `--kv-cache-dtype` 参数：

```bash
# 启用 FP8 E5M2 KV 缓存
python3 -m sglang.launch_server \
    --model-path deepseek-ai/DeepSeek-R1-0528 \
    --kv-cache-dtype fp8_e5m2 \

# 启用 FP8 E4M3 KV 缓存
python3 -m sglang.launch_server \
    --model-path deepseek-ai/DeepSeek-R1-0528 \
    --kv-cache-dtype fp8_e4m3 \

# 启用 FP4 E2M1 KV 缓存
python3 -m sglang.launch_server \
    --model-path nvidia/DeepSeek-R1-0528-NVFP4 \
    --kv-cache-dtype fp4_e2m1 \
```

### 缩放因子

FP8 量化需要缩放因子来正确地量化和反量化 KV 缓存。

```{note}
目前仅支持逐张量（标量）缩放因子。
```

缩放因子可以：

- **从检查点加载**：预量化模型（例如 ModelOpt）可能包含自动加载的 `k_scale` 和 `v_scale` 参数
- **通过 JSON 提供**：通过 `--quantization-param-path` 提供缩放因子。

JSON 文件应遵循以下格式：

```json
{
  "kv_cache": {
    "dtype": "float8_e4m3fn",
    "scaling_factor": {
      "0": {
        "0": 1.0,
        "1": 1.0
      }
    }
  }
}
```

其中 `scaling_factor` 中的外层键是张量并行等级，内层键是层索引。

```{warning}
如果未提供缩放因子且在检查点中未找到，将默认为 1.0，这可能导致精度问题。
```

```{tip}
**FP4 (MXFP4)**：与 FP8 不同，FP4 量化在量化和反量化过程中自动实时处理缩放因子。不需要预量化模型或外部缩放因子文件——基于块的缩放因子在需要时动态计算。
```

## 性能考虑

### 内存节省

量化 KV 缓存提供显著的内存节省：
- **BF16 → FP4**：支持约 3.56 倍于 BF16 的 Token 数量（考虑了缩放因子开销）

```{note}
FP4 和 FP8 量化需要额外的内存来存储基于块的缩放因子，这降低了相对于原始位宽减少的有效内存节省。块大小为 16 的 FP4 支持约 1.78 倍于 FP8 的 Token 数量，约 3.56 倍于 BF16 的 Token 数量。FP8 和 BF16 之间的相对 Token 容量可以从这些比率推导得出。
```

这使得在相同内存预算下可以支持更长的上下文长度或更多的并发请求。

### 精度影响

#### FP8 精度

FP8 E4M3 量化通常引入极小的精度下降。影响取决于模型架构、序列长度和量化格式（通常 E4M3 比 E5M2 具有更好的精度）。

#### FP4 精度

FP4 (MXFP4) 量化提供显著的内存节省，精度影响取决于模型大小和数据集复杂度。来自 [PR #10078](https://github.com/sgl-project/sglang/pull/10078)（MLA）和 [PR #12612](https://github.com/sgl-project/sglang/pull/12612)（MHA）的初步精度测试结果显示：

**大型模型（例如 Qwen3-235B-A22B、DeepSeek-R1-0528）**

在大规模模型上，FP4 保持接近 FP8/BF16 的精度，尤其是在较简单的数据集上：

| 模型 | 数据集 | KV16 | KV8 (FP8 E4M3) | KV4 (FP4 E2M1) |
|-------|---------|------|----------------|----------------|
| Qwen3-235B-A22B | gsm8k | 0.9168 | 0.9181 | 0.9186 |
| Qwen3-235B-A22B | aime25 | 0.7733 | 0.7333 | 0.6000 |
| Qwen3-235B-A22B | gpqa_diamond | 0.7010 | 0.6899 | 0.6778 |
| DeepSeek-R1-0528 | gsm8k | 0.9157 | 0.9154 | 0.9124 |
| DeepSeek-R1-0528 | aime25 | 0.5067 | 0.4934 | 0.4000 |
| DeepSeek-R1-0528 | gpqa_diamond | 0.7707 | 0.7697 | 0.7273 |

**较小模型（例如 GPT-OSS-120B）**

在较小模型上，FP4 表现出更明显的精度下降，尤其是在具有挑战性的数据集上：

| 模型 | 数据集 | KV16 | KV8 (FP8 E4M3) | KV4 (FP4 E2M1) |
|-------|---------|------|----------------|----------------|
| GPT-OSS-120B | gsm8k | 0.9161 | 0.9163 | 0.9152 |
| GPT-OSS-120B | aime25 | 0.7533 | 0.7667 | 0.3533 |
| GPT-OSS-120B | gpqa_diamond | 0.5081 | 0.5434 | 0.3202 |

**关键观察：**

- **简单数据集（例如 gsm8k）**：FP4 在各种模型大小上保持接近 FP8/BF16 的精度
- **模型大小很重要**：大型模型（200B+ 参数）通常比较小模型更能容忍 FP4 量化
- **上下文长度**：在长上下文场景中精度下降可能更加明显，因为量化误差的累积可能变得显著。

```{tip}
请在您的特定模型和工作负载上评估 FP4 精度。大型模型在较简单任务上通常表现出极小的下降，而较小模型或复杂推理任务可能需要 FP8 或 BF16 才能达到可接受的精度。
```

## 最佳实践

- **使用预量化模型**：优先使用在检查点中包含缩放因子的离线量化模型。
- **选择正确的格式**：使用 `fp8_e4m3` 获得更好的精度（推荐），使用 `fp8_e5m2` 获得更大的动态范围，或使用 `fp4_e2m1` 获得最大内存节省（实验性）
- **检查后端兼容性**：验证您选择的注意力后端是否支持量化 KV 缓存
