# 故障排除与常见问题

## 故障排除

本页列出了常见错误和解决方法。

### CUDA 内存不足
如果遇到内存不足 (OOM) 错误，可以调整以下参数：

- 如果 OOM 发生在预填充阶段，尝试将 `--chunked-prefill-size` 降低到 `4096` 或 `2048`。这会节省内存但降低长提示词的预填充速度。
- 如果 OOM 发生在解码阶段，尝试降低 `--max-running-requests`。
- 也可以将 `--mem-fraction-static` 降低到较小值，如 0.8 或 0.7。这会减少 KV 缓存内存池的内存使用，有助于防止预填充和解码阶段的 OOM 错误。但会限制最大并发数和降低峰值吞吐量。
- 另一个常见的 OOM 场景是对长提示词请求输入 logprob，因为这需要大量内存。要解决此问题，在采样参数中设置 `logprob_start_len` 以仅包含必要的部分。如果确实需要长提示词的输入 logprob，请尝试降低 `--mem-fraction-static`。

### CUDA 错误：非法内存访问
此错误可能由内核错误或内存不足问题引起：
- 如果是内核错误，解决可能比较困难。请在 GitHub 上提交 issue。
- 如果是内存不足问题，有时会报告为此错误而非"内存不足"。请参阅上面的章节以避免 OOM 问题。

### 服务器挂起
- 如果服务器在初始化或运行期间挂起，可能是内存问题（内存不足）、网络问题（NCCL 错误）或 SGLang 中的其他 bug。
    - 如果是内存不足，您可能会看到初始化期间或刚初始化后 `avail mem` 非常低。这种情况下，
      可以尝试降低 `--mem-fraction-static`、降低 `--cuda-graph-max-bs` 或降低 `--chunked-prefill-size`。
- 其他 bug，请在 GitHub 上提交 issue。


## 常见问题

### 温度为 0 时结果也不确定

您可能注意到发送两次相同请求时，引擎返回的结果略有不同，即使温度设置为 0。

根据我们的初步调查，这种不确定性来自两个因素：动态批处理和前缀缓存。大致来说，动态批处理占不确定性的约 95%，前缀缓存占其余部分。服务器在底层运行动态批处理。不同的 batch size 会导致 PyTorch/CuBLAS 分派到不同的 CUDA 内核，这会导致轻微的数值差异。这种差异在多层间累积，导致 batch size 变化时产生非确定性输出。类似地，启用前缀缓存时也会分派到不同的内核。

要在当前代码中实现更确定的输出，可以添加 `--disable-radix-cache` 并一次只发送一个请求。在此设置下结果将基本确定。

**更新**：
最近我们还引入了确定性模式，可以通过 `--enable-deterministic-inference` 启用。
更多详情请参见此博客文章：https://lmsys.org/blog/2025-09-22-sglang-deterministic/
